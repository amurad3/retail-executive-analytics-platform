# Power BI Executive Dashboard -- Build Guide

This is a live-build guide, not a script -- Power BI Desktop is a GUI application,
so this dashboard gets built together interactively rather than generated by code.
Everything it needs (the star schema, and six derived `analytics.*` tables) is
already sitting in Postgres, populated by the ETL pipeline and analytics modules.

## Prerequisites

1. **Power BI Desktop** (free) -- install from the Microsoft Store or
   https://powerbi.microsoft.com/desktop/
2. **Npgsql** -- Power BI's PostgreSQL connector requires the Npgsql .NET driver.
   Download the latest `.msi` from https://github.com/npgsql/npgsql/releases
   (look for the Npgsql installer, not just the library) and run it. Without this,
   Power BI's Postgres connector will fail to load.
3. Confirm the local database is reachable: `psql -U postgres -h localhost -d retail_analytics -c "select 1;"`

## Connecting

1. Power BI Desktop -> **Get Data** -> **More...** -> **Database** -> **PostgreSQL database**.
2. Server: `localhost:5432`, Database: `retail_analytics`.
3. Use **Import** mode (not DirectQuery) -- at ~9M/4M rows this comfortably fits
   Power BI's in-memory VertiPaq engine and gives much faster visuals than DirectQuery
   against a single unindexed-for-BI Postgres instance.
4. Credentials: database user/password (`postgres` / value from your local `.env`).
5. In Navigator, select all tables listed below and click **Transform Data** (not
   Load directly) so we can trim a few columns before import.

## Tables to import

| Table | Role |
|---|---|
| `dim_date` | Date dimension |
| `dim_store` | Store dimension |
| `dim_product` | Product dimension |
| `dim_customer` | Customer dimension |
| `fact_sales` | Transaction line-item fact (~9M rows) |
| `fact_inventory` | Monthly inventory snapshot fact (~4.2M rows) |
| `analytics.sales_forecast` | Forecast module output |
| `analytics.customer_segments` | Segmentation module output |
| `analytics.profitability_summary` | Profitability module output (category/region/month grain) |
| `analytics.profitability_by_product` | Profitability module output (product grain) |
| `analytics.inventory_recommendations` | Inventory module output |
| `analytics.product_abc_classification` | Inventory module output (ABC classes) |

In Power Query, on `fact_sales`, consider removing `sale_id` after import if you
don't plan to reference it directly (it's the partition PK component, not needed
for any measure) -- shrinks the model slightly. Everything else can be imported as-is.

## Data model relationships

Power BI should auto-detect most of these; verify in **Model view**:

```
dim_date[date_id]      1 --- * fact_sales[date_id]
dim_store[store_id]    1 --- * fact_sales[store_id]
dim_product[product_id]1 --- * fact_sales[product_id]
dim_customer[customer_id] 1 --- * fact_sales[customer_id]

dim_date[date_id]      1 --- * fact_inventory[date_id]
dim_store[store_id]    1 --- * fact_inventory[store_id]
dim_product[product_id]1 --- * fact_inventory[product_id]

dim_product[product_id] 1 --- * analytics.profitability_by_product[product_id]
dim_product[product_id] 1 --- * analytics.inventory_recommendations[product_id]
dim_product[product_id] 1 --- * analytics.product_abc_classification[product_id]
dim_customer[customer_id] 1 --- * analytics.customer_segments[customer_id]
dim_store[store_id]     1 --- * analytics.inventory_recommendations[store_id]
```

`analytics.sales_forecast` is intentionally **not** related to `dim_date` -- it's
monthly grain (vs. daily for the star schema) and holds both actuals and future
forecast rows past the end of `dim_date`'s populated sales range. Build its chart
directly off `analytics.sales_forecast[month_date]` as a standalone table; let
Power BI auto-generate its own date hierarchy for it.

Set `dim_date` as the model's **Mark as Date Table** (Model view -> right-click
`dim_date` -> Mark as date table -> pick `full_date`), so time intelligence
functions (SAMEPERIODLASTYEAR, DATEADD, etc.) work correctly.

## Suggested report pages

1. **Executive Summary** -- KPI cards (Total Revenue, Gross Margin %, YoY Growth %,
   Return Rate %, Active Customers), revenue trend line by month, revenue by region map/bar.
2. **Sales & Forecast** -- line chart of `analytics.sales_forecast` (actual_revenue
   solid line, forecast_revenue dashed, lower_ci/upper_ci as a shaded band) with a
   category slicer; YoY growth by category bar chart.
3. **Customer Segmentation** -- donut of `rfm_segment` counts, donut of
   `cluster_segment` counts, scatter of recency vs. monetary colored by segment,
   segment table with avg monetary/frequency.
4. **Profitability** -- margin % by category/region matrix or heatmap, top/bottom
   10 products by margin (from `profitability_by_product`), margin trend over time.
5. **Inventory** -- stock status breakdown (Understocked/Overstocked/Aligned) donut,
   ABC classification bar, table of top understocked SKUs by revenue impact.

## Refresh

Since this is Import mode against a local (later Azure) Postgres instance, refresh
manually via **Home -> Refresh** after re-running the ETL/analytics pipeline, or set
up a scheduled refresh once the report is published to the Power BI service and
pointed at the Azure-hosted database (see [infra/azure/README.md](../infra/azure/README.md)).
